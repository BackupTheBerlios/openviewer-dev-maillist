I'm not a fan X-chat UI at all, but I think that its possible to allow
people to easily switch between many smallish virtual worlds as easily
as with the web without making it look like Firefox.

In fact a hybrid of the two UIs, and what I had in mind, might look more
like Pidgin:

- A small session-wide window lists 

  1. The avatars you posses, and the VW they were last logged into. You
click on an avatar and you log into the last know VW. Avatar portability
would make this menu moot.

  2. The distinct grids (SL,HiPiHi,etc) you hold distinct accounts in.
You click on one and you log into it. OpenID would make this menu moot.

  3. A list of "bookmarked" locations, represented by a URI, presented
in menu form. After all there will be less VWs than there are web pages,
and less moving in between them. It makes less sense to force people to
type URIs.

- When a login happens, a 3D display screen comes up -- with tabs if
you're logged into more than one or with more than one avatar.

Cheers,

> > Jeff Ames <jeffames at gmail.com> wrote:
> >  > - Login Screen
> > > - Chat Screen (Local Chat/IM/Group Chat)
> > > - Settings Screen
> >
> > It sounds like you're envisioning a separate tab in the notebook
> > control for each of these functions. I was thinking more along the
> > lines of a Firefox for virtual worlds -- one main viewing window, an
> > address bar, and windows accessible through the menu bar for things
> > like setting preferences, etc. (And perhaps tabbed browsing, with
> > support for multiple virtual world connections simultaneously)
> >
> > One of the major goals of this project, though, is to be highly
> > customizable, so if someone wants to totally rewrite the UI, it's a
> > fairly simple task. We just provide a default implementation (or
> > multiple defaults -- a browser-like skin, a FPS-like skin, etc.)
> >
> > To keep it customizable, we should limit, or at least explicitly
> > define, the dependencies between the .glade file and the code (i.e.,
> > what widgets are expected to exist, and what types and names they
> > should have).
> >
> > Another thing to keep in mind is that at some point in the distant
> > future, the UI may be partially specified by the server, so you can
> > log into a FPS or a virtual conference with the same viewer and get a
> > UI appropriate to each.
> >
> > > - OpenViewer Capabilities Management
> >
> > From what's on the wiki, it sounds like a lot of this should happen at
> > the protocol layer. Then the UI would query the protocol to see if
> > certain features are supported. For the time being, though, I think
> > it's fairly safe to assume that any protocol we're using is a superset
> > of the Second Life protocol.
> >
> > (If anyone wants to add other totally different protocols, they're of
> > course welcome to do so. It would be quite interesting to see how
> > well the IProtocol interface can abstract the differences out.)
> >
> > > - UserSettings
> >
> > This now auto-generates an XML file when you run the app. We may want
> > to add a version header to it, though, to avoid upgrade problems.
> >
> > However, my feeling is that it belongs more at the application layer than
> > in the GTK UI.
> >
> > > - 3D Integration (if not, an external window will be fine)
> >
> > This is something we should look into very soon (though I think MW is
> > already). If we run into problems here (like GTK# not supporting GL
> > contexts, or such), it could seriously limit the usefulness of these
> > UI's.
> >
> > > - fixed some namespacing issues
> >
> > If there are conflicts with the Gtk namespace, we can change to
> > something like GtkUI or GtkPlugin or such. However, the namespace
> > should mirror the file path (so
> > OpenViewer/UI/GtkPlugin/GtkMainWindow.cs should define a class
> > GtkMainWindow in the namespace OpenViewer.UI.GtkPlugin)
> >
> > Also, the way we're using namespaces currently means we can't have
> > classes and namespaces with the same name, so we have to avoid doing
> > something like GtkPlugin/GtkPlugin.cs.
