+1 for Daedius and his thoughts. We are heading into the unknown a bit and our paradigms they will be a shifting a bit.

For one use case, that is a secondlife-type application, a viewer similar in its user interface to secondlife is a good thing.

There will be other use cases that do use URL's as our vision of a 3-D browsing world may come to pass, so perhaps we need to accomodate both these and other notions. 

Maybe a core OpenViewer with plugins using a similar strategy to OpenSim is appropriate. In any case, at this point, I would say, let it evolve and all the personalities on this channel will have an affect on how it looks.

Certainly Mw's is one of the most important voices and I dont want my comments to be construed as negative against those, but I like Daedius thoughts and want to encourage him.

Charles

I don't think the openviewer should look like firefox, and I don't
think it should look like all tabs.

Firstly, I think the address bar is useless.  How often do you think a
person is going to be manually typing in metaverse URLs?  It's only
going to take up space.  90% of the time a user is only going to log
into one or two grids manually and that might be it.  And when they
do, it should be automated or easy for them to log in a second time
without the use of an address bar.  If we wanted to do URL handling of
metaverse URLs ( say when someone clicks something in the REAL
firefox), we can, I just don't think a dedicated addressbar is useful.
 Connection settings are more complex than just URL, and i'm just
plain lazy and wouldn't want to type it out every time or deal with it
in string form.  I think the majority of average users would agree.

I envision open viewer being more like X-Chat.  If any of you are
familiar with that software.

Here is a picture of the main view:
http://farm4.static.flickr.com/3044/2360641855_c22e80af76_o.png  . The
chat area of the screen would be our 3D view area.  I think the area
on the right would make a good communications/inventory area.  And the
area on the left could make a good context sensitive control area like
Michael was mentioning ( for build tools or what not ).  Everything
else would be accessable via menus as Jeff suggested.

Take here and look how they do connections.  Basically, there is a
window that accessable via the menu called Network List.(
http://farm3.static.flickr.com/2397/2361474720_0fa170983c_o.png ) This
is where you manage all your networks you normally connect to, as well
as suggest networks that might be interesting.  I can edit a network
to give it more specific settings  (
http://farm4.static.flickr.com/3225/2360642035_344b729ce9_o.png ).  I
can specify login servers, passwords, certain rooms to join, and set
X-Chat to automatically connect to certain networks on startup.

I imagine we can use X-Chat metaphors very directly.

- Daedius

> Jeff Ames <jeffames at gmail.com> wrote:
>  > - Login Screen
> > - Chat Screen (Local Chat/IM/Group Chat)
> > - Settings Screen
>
> It sounds like you're envisioning a separate tab in the notebook
> control for each of these functions. I was thinking more along the
> lines of a Firefox for virtual worlds -- one main viewing window, an
> address bar, and windows accessible through the menu bar for things
> like setting preferences, etc. (And perhaps tabbed browsing, with
> support for multiple virtual world connections simultaneously)
>
> One of the major goals of this project, though, is to be highly
> customizable, so if someone wants to totally rewrite the UI, it's a
> fairly simple task. We just provide a default implementation (or
> multiple defaults -- a browser-like skin, a FPS-like skin, etc.)
>
> To keep it customizable, we should limit, or at least explicitly
> define, the dependencies between the .glade file and the code (i.e.,
> what widgets are expected to exist, and what types and names they
> should have).
>
> Another thing to keep in mind is that at some point in the distant
> future, the UI may be partially specified by the server, so you can
> log into a FPS or a virtual conference with the same viewer and get a
> UI appropriate to each.
>
> > - OpenViewer Capabilities Management
>
> From what's on the wiki, it sounds like a lot of this should happen at
> the protocol layer. Then the UI would query the protocol to see if
> certain features are supported. For the time being, though, I think
> it's fairly safe to assume that any protocol we're using is a superset
> of the Second Life protocol.
>
> (If anyone wants to add other totally different protocols, they're of
> course welcome to do so. It would be quite interesting to see how
> well the IProtocol interface can abstract the differences out.)
>
> > - UserSettings
>
> This now auto-generates an XML file when you run the app. We may want
> to add a version header to it, though, to avoid upgrade problems.
>
> However, my feeling is that it belongs more at the application layer than
> in the GTK UI.
>
> > - 3D Integration (if not, an external window will be fine)
>
> This is something we should look into very soon (though I think MW is
> already). If we run into problems here (like GTK# not supporting GL
> contexts, or such), it could seriously limit the usefulness of these
> UI's.
>
> > - fixed some namespacing issues
>
> If there are conflicts with the Gtk namespace, we can change to
> something like GtkUI or GtkPlugin or such. However, the namespace
> should mirror the file path (so
> OpenViewer/UI/GtkPlugin/GtkMainWindow.cs should define a class
> GtkMainWindow in the namespace OpenViewer.UI.GtkPlugin)
>
> Also, the way we're using namespaces currently means we can't have
> classes and namespaces with the same name, so we have to avoid doing
> something like GtkPlugin/GtkPlugin.cs.
