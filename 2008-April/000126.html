I have worked quite a lot with Test Driven Development.
 
+1 on doing it now, if it's ever going to be done; retrofitting code for TDD is notoriously hairy.
 
* Make sure everbody sees the difference between unit testing and integration testing (and that they should be separate test spaces)
* Badly written tests are as hampering for development as badly written code. Make sure your test test exactly the requisite it's supposed to; not more, not less. Refactor code to accomodate for testing.
* Tests should live in assemblies side by side with the code it's testing, following the same code and name conventions.
 
One of the effects of TDD is 'decoupling', since you want to be able to test isolated units.
Also, you really don't want your call stack to get deep (since that's complicating the scenario setup) so code for simple operations involving few actors.
Also, TDD tends to push the developer away from state; it's difficult to write tests to proof changes in the internal state of an object. Incidentally, this leads to better code, as many bugs and unnessecary complexities emanate from use of state.
/Stefan
